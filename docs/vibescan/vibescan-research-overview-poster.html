VibeScan: Static Security for AI-Assisted Node.js & TypeScript

Research-driven static analysis with explicit rules, reproducible scans, and pipeline-ready signals.

Open-source scanner (npm: vibescan) — static rules for crypto & injection, Express heuristics, LLM/chatbot integration code patterns, optional npm audit merge, optional shallow HTTP GET probes, and LLM-ready prompt export — complementary to model red-teaming and org policy layers (secure-arch).

1. Research Motivation & Core Claim

Recent work reports uneven security quality in AI-assisted backends and risks such as weak defaults and risky dependency choices. Prompting alone is not enough.

Core claim: downstream static analysis can act as a practical safety layer for AI-assisted development by improving repeatability, auditability, and CI enforcement.

See docs/vibescan/abstract.md for citation anchors (e.g., SusVibes, BaxBench, USENIX Security 2025 slopsquat discussion).

Research question

Can a bounded, rule-explicit static analyzer improve repeatability, auditability, and CI enforcement for Node/Express-style JavaScript — especially alongside AI-generated code — compared to ad hoc review or single-tool baselines?

2. Features, Coverage & Architecture

Rule-based & taint checks

AST rules for weak crypto, injection sinks, JWT/secret misuse, plus taint-style source→sink flows where implemented.

LLM/chatbot heuristics

Static checks for common LLM-app failures: dynamic system prompts, RAG/template boundary mixing, unsafe DOM sinks for model output — not jailbreak scoring.

Express route intelligence

Heuristics for missing auth on sensitive routes, rate limits, webhook verification gaps, and related middleware patterns.

CI outputs

JSON and SARIF; non-zero exit on critical/error. Optional policy-check / policy eval scripts in-repo.

Registry + npm audit

--check-registry → SLOP-001 (slopsquat-style). --npm-audit merges advisory findings as supply_chain.npm_audit (same run / SARIF). Data still comes from npm.

Optional HTTP probes

--http-probe-url issues shallow GETs for discovered Express routes (status/latency). Not DAST, auth flows, or session testing.

Fix prompt export

--fix-prompt-file exports a deterministic Markdown remediation prompt from findings for ChatGPT/Copilot/etc. No model call in-tool; pairs with static remediation text.

Generated test stubs

Optional --generate-tests to scaffold security-oriented tests (starting points for human review).

Architecture: deterministic coverage pipeline

Clone / project inputs→ Parse + taint→ Rules + Express heuristics + LLM checks→ Evidence outputs (JSON/SARIF) + fix prompt

Designed for versioned reruns and CI gate checks, not best-effort LLM judgments.

Workflow

Install vibescan (npx / npm i)→ vibescan scan (or secure scan)→ Rules & config (severity threshold)→ Optional checks (--check-registry, --npm-audit, --http-probe-url)→ Artifacts: JSON + SARIF→ CI gate: non-zero on critical/error→ Optional remediation: fix-prompt / test stubs

The npm package is the scanner; this poster’s monorepo includes benchmarks + demo UI. Artifacts are designed for CI merge-gate workflows.

AI Positioning & Honest Limits

VibeScan does not run prompts or measure jailbreak success — it analyzes JavaScript/TypeScript for patterns that often appear in LLM-backed apps and coding-agent output.

  • Static coverage for LLM-app code planes: injection.llm.dynamic-system-prompt (non-literal system content), injection.llm.rag-template-mixing (retrieval + template boundaries), injection.llm.unsafe-html-output (model output into innerHTML-style sinks / XSS class).
  • Connector abuse overlaps classic rules: command injection, eval, SSRF-style flows, and weak auth on sensitive routes — relevant when agents call tools or admin APIs.
  • What stays outside the scanner: dataset poisoning, memorization/PII probing, and full prompt-injection & jailbreak pass/fail. Use red-team suites, output filters, human approval on tools, and monitoring. This repo includes benchmarks/llm-redteam/ for evaluation-style work; full mapping: docs/vibescan/llm-threat-coverage.md.
  • secure-arch (monorepo): a policy/architecture layer that complements VibeScan with YAML guardrails and IDE adapters (docs/secure-arch/README.md).

Evidence & limitations

  • Preliminary DVNA-style evidence under benchmarks/results/ — not full precision/recall.
  • Gold-path + unit tests cover scan regression; npm-audit JSON parsing, HTTP probe, and fix-prompt builders have dedicated tests in vibescan/tests/unit/.
  • Comparison harness aligns VibeScan vs ESLint-style signals on shared evidence surfaces; deeper SAST baselines (e.g., Bearer/Snyk) are not yet frozen in the same environment.
  • secure-arch is related (policy layer), but the core evaluated contribution here is the deterministic scanner.
