VibeScan — Static security for JavaScript & TypeScript

Research-driven tooling for AI-assisted backends: explicit rules, reproducible scans, and pipeline-ready signals.

Open-source scanner (npm: vibescan) focused on OWASP-style crypto & injection classes, Express-oriented heuristics, and evaluation harnesses in this monorepo — not a full application security platform.

Research motivation

Recent work reports uneven security quality in LLM-generated backends and risks such as weak defaults and risky dependency choices. That motivates deterministic, versioned checks in the loop — not prompt tweaks alone.

See docs/vibescan/abstract.md for citation anchors (e.g., SusVibes, BaxBench, USENIX Security 2025 slopsquat discussion).

Research question

Can a bounded, rule-explicit static analyzer improve repeatability, auditability, and CI enforcement for Node/Express-style JavaScript — especially alongside AI-generated code — compared to ad hoc review or single-tool baselines?

What we ship (features)

Rule-based & taint checks

AST rules for weak crypto, injection sinks, JWT/secret misuse, plus taint-style source→sink flows where implemented.

Express route intelligence

Heuristics for missing auth on sensitive routes, rate limits, webhook verification gaps, and related middleware patterns.

CI outputs

JSON and SARIF; non-zero exit on critical/error. Optional policy-check / policy eval scripts in-repo.

Dependency signals

Optional --check-registry for slopsquat-style package-name signals (SLOP-001). Complements npm audit, does not replace it.

Generated test stubs

Optional --generate-tests to scaffold security-oriented tests (starting points for human review).

Benchmarks & demo

Gold-path vulnerable/fixed pairs, scan regression tests, comparison harness vs ESLint/npm audit/optional CLI baselines, interactive demo UI.

How we work (project workflow)

Rule design & CWE mapping→ TypeScript implementation→ Unit tests (scanner)→ Gold-path / DVNA-style fixtures→ Adjudication docs→ npm publish vibescan
Clone / CI→ vibescan scan→ Findings + SARIF/JSON→ Policy gate (optional)→ Fix & re-scan

Monorepo: scanner under vibescan/; research & benchmarks under docs/vibescan/, benchmarks/, demo/. Claims stay conservative until multi-baseline metrics are frozen (see abstract.md).
