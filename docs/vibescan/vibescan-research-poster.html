VibeScan: Security Verification and Pipeline Protection for AI-Assisted JavaScript Applications

AI-generated backend code can look functional while still shipping weak secrets, unsafe dependencies, missing auth checks, and other security flaws.

VibeScan is a JavaScript/TypeScript security tool that combines static analysis, benchmark-ready evidence, and pipeline protection to catch risky patterns before deployment.

Why this matters

  • LLM-generated backend code has inconsistent security quality even when it appears correct.
  • Hallucinated npm package names and weak defaults create real supply-chain and deployment risk.
  • Teams need repeatable safeguards, not just one-time prompt fixes.

Research question

Can a deterministic security verification tool provide clearer coverage, more repeatable outputs, and better pipeline protection for AI-assisted Node/Express code than prompt-only or ad hoc security workflows?

Core contributions

  • Static JS/TS security scanner for Node/Express-style apps.
  • Route and middleware analysis beyond basic payload checks.
  • Optional registry check for hallucinated npm dependencies (--check-registry / slopsquat-style signals).
  • Benchmark and adjudication export for reproducible evaluation.
  • Policy-driven CI enforcement concept (JSON/SARIF → merge gate).

What VibeScan is

Static analysis

Crypto patterns · injection (SQL/command/eval) · auth/middleware heuristics · OpenAPI drift / route inventory

Dependency trust

Slopsquat-style signals · optional npm registry verification

Verification support

Generated test stubs · benchmark + adjudication export

Pipeline protection

JSON / SARIF output · policy checks · CI fail conditions

Verification / pipeline story

Scan→ Findings→ JSON / SARIF→ Policy check→ Security tests→ Merge gate

VibeScan is evolving from a static scanner into a security verification and pipeline protection tool with explicit, repeatable coverage.

Why this is better than AI-only workflows

VibeScan makes coverage explicit and enforceable; AI assistance may be helpful but is less deterministic.

Method / evaluation setup

  • Benchmark target: DVNA (appsecco/dvna), first-party code only.
  • Ground truth: manual adjudication of counted true positives.
  • Baseline run: eslint-plugin-security (pinned config in repo).
  • Additional baselines (Bearer, Snyk Code) are documented but not run in the same environment yet.
  • Results are preliminary and scope-limited — see benchmarks/results/legacy/dvna-evaluation.md.

Real-world implications

  • Helps small teams shipping AI-assisted apps without dedicated security review.
  • Reduces risk around identifiers, payments, and account data.
  • Improves trust in software pipelines through repeatable checks.

Limitations

  • Benchmark is preliminary and incomplete.
  • Bearer and Snyk were not run in the same environment yet.
  • Static analysis is heuristic, not proof of security.
  • Full TP/FP/FN precision/recall remains future work.

Future work

  • Complete multi-baseline benchmark (Bearer, Snyk, frozen versions).
  • Expand generated security tests.
  • Strengthen policy-check / merge-gate workflow.
  • Add more seeded benchmark cases.
  • Compare against AI-assisted security workflows more formally.
