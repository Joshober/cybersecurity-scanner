Motivation

Vibe coding lets developers ship full-stack apps from natural language, but systemic security failure follows: wrong dependencies, default secrets, and missing auth checks slip into production. Prompt engineering alone cannot close that gap.

Approach

VibeScan runs downstream static analysis on generated Express/Node code: AST route graph, taint-style checks, registry-backed slopsquatting detection, and generated .test.js security probes—without requiring a formal spec from the author.

Novel contributions

01
Spec-Free AST Route Graph

Extracts mounted routes and handlers from idiomatic Express without an OpenAPI spec.

02
Persistent Security Test Files

Emits .test.js files that encode exploit hypotheses for CI, not one-off chat fixes.

03

RULE-SLOP-001 verifies dependencies against the public npm registry before ship.

Lessons learned

  • Security prompts trade quality for little gain. SusVibes shows security-focused prompting degrades functional correctness without reliably improving security (arXiv:2512.03262).
  • Passing the benchmark is not enough. BaxBench reports ~50% of backends that pass automated checks remain exploitable (arXiv:2502.11844).
  • Models leak predictable default secrets. Invicti found 1,182 apps using the same placeholder key—each LLM has a fingerprintable default-secret vocabulary (2025).
  • Nearly one in five package suggestions are fake. Spracklen et al. measure 19.7% hallucinated npm names in LLM outputs—supply-chain risk is structural (USENIX Security 2025).

Architectural lesson: Ship downstream scanning in the build pipeline—not more prompt engineering—as the primary safety layer for vibe-coded systems.

Tenzai (Jan 2026): 100% SSRF detection vs 0% CSRF in evaluated AI-generated stacks—coverage is uneven; static tools must fill the holes.
